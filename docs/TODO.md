# 本番運用ロードマップ（移行フロー）

本サービスを本番運用・規模拡大・**メインサイトの録画(録音)**まで進めるための作業を、
**到達順(フェーズ)に並べた移行フロー**として記録する。各フェーズは独立して価値が出る単位で、
「いつ着手するか(トリガ)」を明記している。背景の詳細は CHANGELOG / SPEC を参照。

到達点の要約: **ステートレスFastAPI×N台 + Redis(待機列/WS/統計/設定) + PostgreSQL + TURN +
メインサイトのSFU(LiveKit)で音声録画→オブジェクトストレージ + ジョブ基盤**。
サブサイトは P2P + TURN を継続。**「録画」がSFU採用を決める分岐点(Phase 2)**で、ここで
録画とメディアのスケールを同時に取り、自前WebRTCシグナリングを廃止できる。

```
Phase0(=デモ完成形)     Phase1            Phase2(録画/分岐)        Phase3(スケール)      Phase4
P2P + TURN + Postgres → Redis一部/ジョブ → メインSFU+録画         → 状態を全Redis化     → 最適化
単一インスタンス         単一(下地づくり)   メイン=SFU / サブ=P2P    ステートレス・N台・HA   地理分散/DR
```

---

## アーキテクチャ前提（現状・重要）

- **現状は「単一プロセス・単一インスタンス」前提**。以下を**プロセス内メモリ**で保持しており、
  ワーカー/インスタンスを増やすと待機列・通話・統計・設定が分断して破綻する:
  - マッチング待機列・WebSocket接続・通話セッション（`app/matching.py` の `MatchingManager`）
  - リアルタイム統計のブロードキャスト（配信先は自プロセスの接続のみ）
  - サイト設定キャッシュ（`app/main.py` の `_settings_cache`）
- **当面の運用ルール**: スケール設定は **インスタンス数=1・ワーカー数=1** に固定（`render.yaml`/`Dockerfile`
  の uvicorn 1ワーカーは条件を満たす）。この制約を外すのは **Phase 3**。

## Phase 0 — 永続化 + TURN（= デモの完成形・今すぐ）

**目的**: 「データが消える」「通話が繋がらない」という本番ブロッカー2つを潰す。単一インスタンスのまま。
**トリガ**: 今すぐ。 **工数**: 小。

- [ ] **データベースの永続化（SQLite → PostgreSQL）** — 現状SQLiteは再デプロイ/インスタンスごとに消え、
      正規ユーザーも毎回ログアウトされる（`token_version`で「別人ログイン」は防止済みだが恒久対応はDB永続化）。
      `DATABASE_URL` 対応は実装済み。マネージドPostgresを用意し `DATABASE_URL=postgresql://...` を設定、
      `psycopg2-binary`(or `psycopg`)を requirements に追加。Renderなら永続ディスク+SQLiteでも当座は可だが、
      Phase 3を見据えるとPostgres推奨。
- [ ] **TURNサーバーの導入（マネージド）** — 現在はGoogle STUNのみで、企業NW/モバイル(対称NAT・CGNAT)で
      P2P失敗。マネージドTURN(Cloudflare Calls / Metered / Twilio など)を採用し、`turns:443/tcp` を含める。
- [ ] **TURN credentialの安全な配布** — JSに直書きせず、`/api/turn-credentials` を新設して**短期credential**を
      サーバー発行。`static/app.js` の静的 `RTC_CONFIG` を、通話開始時にサーバーから取得する方式へ変更。
- [x] **SECRET_KEYの設定** — Renderは `render.yaml` の `generateValue: true` で安定値が自動設定済み
      （既定値 `dev-secret-change-me` はローカルのみ）。他環境へ移す場合のみ環境変数で設定すること。
- [ ] **SMTP設定（任意・運用開始時）** — `SMTP_HOST/PORT/USER/PASS/FROM`。未設定の間は招待・リセット・
      チーム予定通知メールは送信されず画面表示のみ（コードは送信スキップで安全に動作）。

## ★ ローンチ前に必須（フェーズ非依存のハードニング）

**目的**: 一般公開する前に必ず閉じるセキュリティ/デモ撤去。Phase 0と並行で実施し、公開のゲートにする。
**トリガ**: 一般公開の直前。

- [ ] **administratorの初回パスワード変更強制を復活** — デモ版で無効化中（`app/main.py` の `seed_initial_data`）。
      `must_change_password=True` に戻し、既定パスワード `password` を廃止。
- [ ] **ログイン試行回数制限（ブルートフォース対策）** — slowapi等でIP/ユーザー単位のレート制限。
- [ ] **CORSの `allow_origins` を本番ドメインに限定**。
- [ ] **自己登録機能の廃止** — 本番はメインサイトも管理者作成のみの想定。
- [ ] **初期パスワード運用ルールの整備** — 現状は初回ログイン時の変更強制のみ。
- [ ] **デモデータ生成・削除機能の撤去** — 開発者ページの「デモデータ」パネルと
      `POST/DELETE /api/dev/demo-data`（`create_demo_data`/`delete_demo_data`）を丸ごと削除。
      ※現在は実APIルート経由で生成/削除（DB直挿入はマッチング記録 `CallPair` のみ）。

## Phase 1 — スケール下地（挙動を変えずに準備）

**目的**: Phase 2/3に向けた土台づくり。まだ単一インスタンスだが、外部化・抽象化を先行する。
**トリガ**: 録画 or 多インスタンス化が視野に入ったら。 **工数**: 中。

- [ ] **Redis導入（影響の小さい状態から）** — まず**設定キャッシュ・統計・オンライン/プレゼンス**をRedis化
      （pub/subで無効化通知）。マッチング待機列はまだプロセス内のまま（Phase 3で移す）。
- [ ] **ジョブ基盤の導入** — メール送信・予定リマインド・録画後処理・保持期間削除などを
      `BackgroundTasks`(再起動で消える)から **arq/Celery(Redis)** か Cloud Tasks へ移管。
- [ ] **メディア層の抽象化** — 通話確立処理をインタフェース化し、後で P2P→SFU を差し替えやすくする。
- [ ] **可観測性** — 接続成功率・relay(TURN)使用率・WS接続数などのメトリクス/ログ/トレースを計測開始。

## Phase 2 — 録画（メインサイトにSFU導入）★分岐点

**目的**: メインサイトの**音声録画**を実現。同時にメインサイトのメディアをサーバー経由にしてスケール可能化。
**トリガ**: 録画を商用機能として提供すると決めたとき。 **工数**: 大。

- [ ] **SFU(LiveKit)導入** — マッチング成立後、2人をLiveKitルームに参加させる。
      これにより `app/matching.py` のWebRTCシグナリング中継(offer/answer/ICE)を**撤去**でき、
      TURNもSFU側に集約される。**サブサイトはP2P+TURNのまま**（録画不要・コスト最小）。
- [ ] **録画(egress)** — 音声を録音（実映像モードのみ映像も可）。**アバター/静止画モードはサーバー上に
      音声しか存在しない**点に注意（アバター映像はクライアント描画）。
- [ ] **同意・保持・権限の設計（録画の前提）** — 双方の**明示同意**＋録画中インジケータ、保持期間と
      自動削除、再生できる人の制御（本人/ T&S目的のモデレータ等）、保管時暗号化、法令対応。
- [ ] **オブジェクトストレージ** — S3/GCSに保存、ライフサイクルで自動失効。メタデータはPostgres
      （call_id・参加者・長さ・保管キー・同意・失効日）。

## Phase 3 — 水平スケール（ステートレス化）

**目的**: 無停止デプロイ・高可用・同時接続上限の突破。
**トリガ**: HAが必要 / 同時接続が1インスタンスで苦しい（数千WS・CPU が目安）。 **工数**: 大。

- [ ] **マッチング待機列とWS配信をRedis(pub/sub)へ外部化** — `MatchingManager` のプロセス内シングルトンを解体。
      残るP2P(サブサイト)のシグナリング中継もRedis pub/subでインスタンス跨ぎに。
- [ ] **アプリのステートレス化 + 複数インスタンス + LB**。設定キャッシュもpub/sub無効化で整合。
- [ ] **PostgreSQLの接続プール(pgbouncer)** とコネクション設計の見直し。
- [ ] **アーキテクチャ前提の解除** — 上記完了後、「単一インスタンス運用」の制約を外す。

---

## Phase 4 — 最適化・拡張

**目的**: 規模・地理分散・コスト最適化・運用堅牢化。
**トリガ**: スケール後の改善フェーズ。 **工数**: 継続。

- [ ] **リージョン別SFU / TURN** で遅延最小化。
- [ ] **録画のライフサイクル自動化**（保持・削除・監査・コスト監視）。
- [ ] **P2P / SFU のサイト別使い分けの最適化**（保守負担が増えたらSFU統一も再評価）。
- [ ] **バックアップ / DR / 障害復旧手順**の整備。
- [ ] **TURN/SFU を自前運用へ移行**するか再検討（従量課金が固定費を上回る規模になったら coturn / LiveKit自ホスト）。
